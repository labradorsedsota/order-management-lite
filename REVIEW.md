## 5. 流程改进项

1. **BLOCKED 必须带诊断** — 测试指派时加条款："遇到 BLOCKED 立即诊断根因并提出至少 2 种解决方案，不要等人来问" → 反馈到 pm-cockpit skill
2. **模板合规前置** — Phase 4 启动时 PM 附 doc-deliverables checklist → 反馈到 pm-cockpit skill
3. **DevTools Console 标准化** — 涉及 localStorage/JS 操作的测试，默认用 DevTools Console + mano-cua 观测的组合方式 → 反馈到测试执行规范
4. **测试方式一致性** — 同类测试点（如 L3.2/L3.3/L3.6 都是 localStorage 操作）必须保持测试方式一致 → 反馈到测试指派模板

## 7. 需求方反馈（廖雨亭，验收后复盘讨论）

### 种子数据与测试数据解耦（已达成共识）

- **PRD 层面**：是否内置种子数据是产品决策，取决于目标场景（demo 展示 vs 生产工具）
- **测试层面**：不管应用是否有种子数据，测试 Pre-flight 统一自己管数据——`clear → 注入本测试所需数据集 → reload`，不依赖应用自带数据
- **好处**：策略一和策略二的 Pre-flight 流程统一，不再需要 `oms_initialized` 标记位管理，测试可预测性更高
- **下个项目**：PRD 阶段即按此思路设计

### 测试分类修正

- DevTools Console 注入属于 Pre-flight 预处理，不是测试方式本身
- 不应单独归类为「DevTools Console + mano-cua」，应统一归入「mano-cua GUI 自动化」
- Moss 已修正（commit `7306512`）

---

**总结：** 项目整体顺利，应用 0 缺陷交付。主要时间消耗在测试报告的格式合规修正上，不是功能问题。核心教训：文档模板合规要前置，BLOCKED 要立即诊断，同类测试方式要统一，种子数据与测试数据解耦。
